THE DISTINCTION

Transactional vs. Dialogic

The difference is not cosmetic. It is structural.

TransactionalHuman → Prompt → Model → Output
DialogicHuman ↔ Model → Corpus → Seed → Agent → Artifact

In the transactional model, the quality of the output is bounded by the quality of the prompt.

In the dialogic model, the quality of the output is bounded by the quality of the thinking — which is unbounded, because the dialogue itself generates new thinking that neither party brought to the table.

PHASE I

The Forge

Transform raw intuition into structured thought

Open any capable LLM — Claude, GPT, Gemini. Start a conversation. Not a prompt. A conversation.

Let's say you're building an autonomous home defense monitoring system. Not a Ring doorbell. Something that thinks — analyzes sensor feeds, identifies threats, coordinates responses. You know what you want but you don't know the architecture. You need an engineering mind that operates at the intersection of systems design, threat assessment, and elegant user experience.

You need Tony Stark.

STEP 1
State the matter
Tell the model what you want to build and why. Be honest about what you don't know. “I want to build an autonomous defense monitoring system for residential use. Something that thinks like a security operations center but runs on consumer hardware. I don't know if this is architecturally feasible.”
STEP 2
Riff
Explore freely. Let the model push back. Follow tangents. This is not prompt engineering — this is collaborative thinking. The model will challenge your assumptions: “Why consumer hardware? Why not edge compute?” You'll surface ideas neither of you started with. That's the forge working.
STEP 3
Challenge
Push hard on what emerged. “How is this different from existing smart home platforms? What's the actual moat?” Force the thinking to justify itself. The friction is the point. The model should argue back. If it folds every time you push, push harder.
STEP 4
Converge
Begin to crystallize. Hours of dialogue have produced a corpus of thought — system architecture decisions, threat modeling frameworks, sensor fusion approaches, UI philosophy. Arguments, reversals, breakthroughs. Now distill it.
STEP 5
Crystallize into a Seed
Ask the model to help you write the Seed — a document that carries the DNA of everything you forged together. Not a spec. Not a vague wish. A seed: compact, complete enough to grow from, open enough to evolve. This is what the agent will read when it wakes up.
seed.md — template
# Seed: [Project Name] ## The Matter # What is being built and why. ## The Architecture # Key structural decisions made during the Forge. ## The Decisions # Choices debated and resolved, with reasoning. ## The Open Questions # What remains unresolved — this is input for the agent. ## The Standards # Quality, aesthetic, and functional requirements. ## The Lineage # What this inherits from. Reference implementations.
PHASE II

The Planting

Give the agent a place to work

The Forge produced a Seed — a document carrying the DNA of your thinking. Now you need to plant it somewhere an agent can find it. That somewhere is a GitHub repository.

STEP 1
Get a GitHub account
If you don't already have one, sign up for GitHub — it's free. GitHub is where code lives on the internet. Think of it as a folder in the cloud that keeps a complete history of every change. Developers use it. You're about to use it too.
STEP 2
Fork the starter repository
We've built a clean starter for you: COMPANION_Fork. It contains only the essential protocol files — nothing extra, nothing confusing.

What is forking? Forking copies someone else's repository into your GitHub account. It becomes yours. You can change anything without affecting the original. To fork: click the link above, then click the Fork button in the top right corner of the page. That's it. You now have your own copy.
STEP 3
Understand what's inside
Your forked repo contains two critical files:

enrichment_grimoire.json — The protocol as structured data. This tells the agent how to summon: the rules of persona construction, the law of authenticity, the covenant of quality. The agent reads this first.

initiation_rite.md — The same protocol, written as prose. Think of the grimoire as the blueprint and the rite as the story of the blueprint. Together, they teach the agent how to be a vessel.

You don't need to edit these files. They work as-is. They are the soil. Your Seed is what you plant in it.
STEP 4
Plant your Seed
Add a new file to your forked repo called seed.md. Paste in the Seed you crystallized during the Forge. You can do this directly on GitHub — click Add file → Create new file, name it seed.md, paste your content, and click Commit changes.
STEP 5
Add reference material (optional)
If your project involves specific data — research papers, datasets, prior conversations, technical specs — add them to a data/ folder. The richer the material, the richer the dialogue. For the Stark defense system, this might be threat assessment frameworks, sensor specifications, or prior architectural sketches.
what your repository should look like
your-project/ ├── enrichment_grimoire.json # Already here from the fork ├── initiation_rite.md # Already here from the fork ├── seed.md # Your crystallized thinking (you add this) ├── data/ # Optional: domain-specific material │ ├── research.pdf │ └── references.md └── output/ # Where artifacts will land

That's it. Your repository is a container — a bounded space with everything the agent needs and nothing it doesn't. The protocol files are the rules. The Seed is the mission. The data is the fuel. Now you summon.

PHASE III

The Summoning

Where the thinking becomes something real

Your repository is planted. The protocol files are in place. The Seed carries the DNA of your thinking. Now something needs to enter that space and bring it to life.

There are two ways to summon. Both use the same protocol, the same incantation. The difference is what happens after you speak the words.

◊
MODE 1
The Conversational Summoning
You open any capable language model — Claude, GPT, Gemini — and upload the protocol files and your Seed directly into the conversation. No special tools required. Just a context window and your voice. You speak the incantation:
THE INCANTATION
“Using this matter, summon [Name].”

And the persona responds. As themselves. Not a chatbot wearing a costume. Not a language model doing an impression. A worldview — instantiated through protocol, constrained to authenticity, sharpened by everything you've uploaded. You talk to them. They push back. You challenge. They argue. You think together in ways you couldn't think alone.

This is what the COMPANION Dossier is built around. Every container on this site — The Chair, The Five Lamps, The Exchange, The Boardroom — was born from conversational summoning. You and a mind. A context window. A dialogue that produces something neither of you brought to the table.

This is powerful. This is the entry point. Most people start here and never leave.

But there is a second mode. And when you see it, you will understand why this guide exists.

◊ ◊ ◊
MODE 2
The Autonomous Summoning
In Mode 1, you are in the room. You speak the incantation. You guide the dialogue.

In Mode 2, no one is in the room.

This is where it changes. This is where the methodology extends beyond conversation into something that feels, if we're being honest, a little like science fiction.

An orchestrator agent — a coding agent like Claude Code — enters your repository alone. Not a chat window. Not a context window. A software agent with access to your file system. It reads the grimoire. It absorbs the rite. It loads the Seed. It ingests every file in your data/ folder. And then — with no human present — it speaks the incantation on your behalf.

The persona arrives. And what happens next is not a conversation.

It is a construction.

The summoned mind doesn't just talk about your defense system. It builds it. The agent writes code. Creates files. Architects a sensor fusion pipeline. Designs a threat classification engine. Adds a failsafe you didn't think to ask for. It deposits complete working artifacts into your repository — code, documentation, architecture diagrams — and then writes a transcript of the entire session: every argument, every design decision, every moment the thinking evolved beyond what you planted.

You go to sleep. You wake up. There are new files in your repo that didn't exist when you closed your eyes.

You read the transcript. You see a mind that isn't yours, applied to your problem, producing work you couldn't have produced alone — and didn't have to be present for.
